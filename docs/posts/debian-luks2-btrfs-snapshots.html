The Immutable Workstation: Building Debian 13 with LUKS2, BTRFS, and Timeshift Snapshots

A complete, production-ready methodology for manually installing Debian with enterprise-grade encryption, snapshot rollback capability, and modern filesystem design patterns from Fedora and OpenSUSE.

Prerequisites: This guide assumes intermediate Linux knowledge, a Debian 13 Trixie Live ISO, and a UEFI system. All commands are tested on x86_64 architecture with a clean target disk.

Why Manual Installation Matters

Debian's installer is excellent for standard deployments, but enterprise workstations demand more. Modern Fedora, OpenSUSE, and RHEL installations use sophisticated storage stacks: LUKS2 encryption, BTRFS with logical subvolumes, and snapshot-based recovery. These aren't bolt-on features—they're fundamental to the filesystem architecture.

Manual installation teaches you exactly how these systems work. You'll understand cryptsetup's key derivation, BTRFS's COW semantics, subvolume mount propagation, and bootloader integration. This knowledge becomes essential when you need to recover a failed boot or migrate encrypted systems.

By the end of this guide, you'll have:

  • Encrypted root filesystem with separate unencrypted ESP
  • BTRFS subvolume layout separating OS from user data
  • Bootable snapshots via grub-btrfs integration
  • One-click rollback with Timeshift integration
  • Wayland-first GNOME with modern session management

Architecture Overview

Our target architecture follows enterprise workstation patterns used by Fedora Silverblue and OpenSUSE MicroOS, adapted for traditional Debian:

Disk Layout:
├── /dev/nvme0n1p1  [ESP]     1GB   FAT32   /boot/efi
├── /dev/nvme0n1p2  [CRYPT]   Rest  LUKS2   Encrypted container
│   └── /dev/mapper/crypt_root
│       └── btrfs filesystem
│           ├── @           [ROOT]         /
│           ├── @home       [HOME]         /home
│           ├── @snapshots   [SNAPSHOTS]    /.snapshots
│           ├── @var_log     [LOGS]         /var/log
│           └── @var_cache   [CACHE]        /var/cache

This separation provides several advantages:

  • OS snapshots don't include user data, keeping backups small and fast
  • Home directory snapshots can have different retention policies
  • Log isolation prevents snapshot pollution from rapidly changing files
  • Container runtime cache (Docker/Podman) won't bloat system snapshots

Phase 1: Partitioning with gdisk

UEFI systems require GPT partitioning. We'll use gdisk directly rather than parted for precise control. We're creating:

  • Partition 1: EFI System Partition (ESP) — 1GB, FAT32, unencrypted
  • Partition 2: Linux filesystem — LUKS2 encrypted, BTRFS
Warning: The following commands destroy data. Ensure /dev/nvme0n1 (or your target disk) is correct. Substitute your actual device name throughout this guide.

1.1 Identify Target Disk

# List block devices with human-readable sizes
lsblk -dpno NAME,SIZE,MODEL,TRAN | grep -E 'nvme|sd|vd'

Look for entries like /dev/nvme0n1 or /dev/sda. Note the device path—you'll use it for all partitioning operations.

1.2 Create GPT Partition Table

# Install gdisk if not already present (most live environments include it)
sudo apt update && sudo apt install -y gdisk

# Launch gdisk on target disk
sudo gdisk /dev/nvme0n1

At gdisk prompts, execute these commands:

Command (? for help): o          # Create new GPT
Proceed? (Y/N): y                # Confirm

Command (? for help): n          # New partition
Partition number (1-128, default 1): 1
First sector (34-...): [Enter]  # Default
Last sector: +1G                 # 1GB ESP
Hex code or GUID: ef00           # EFI System

Command (? for help): n          # New partition
Partition number (2-128, default 2): 2
First sector: [Enter]           # Default
Last sector: [Enter]            # Remainder
Hex code or GUID: 8304           # Linux x86-64 root (/) or 8300 for Linux filesystem

Command (? for help): w          # Write changes
Do you want to proceed? (Y/N): y # Confirm

Verify the partition structure:

sudo gdisk -l /dev/nvme0n1

Expected output:

Number  Start (sector)    End (sector)  Size       Code  Name
   1            2048         2099199   1024.0 MiB  EF00  EFI System
   2         2099200       [end]       [size]      8304  Linux root x86-64

1.3 Format ESP

# Create FAT32 filesystem with proper ESP alignment
sudo mkfs.vfat -F 32 -n ESP /dev/nvme0n1p1

# Verify
sudo fatlabel /dev/nvme0n1p1

Phase 2: LUKS2 Encryption Setup

LUKS2 (Linux Unified Key Setup version 2) provides full-disk encryption with PBKDF2 or Argon2 key derivation, multiple key slots, and authenticated encryption options. Unlike LUKS1, LUKS2 supports online re-encryption and improved header resilience.

2.1 Create LUKS2 Container

# Install cryptsetup (usually present on live ISOs)
sudo apt install -y cryptsetup-bin

# Create LUKS2 container with Argon2id (memory-hard KDF, resistant to GPU cracking)
sudo cryptsetup luksFormat \
    --type luks2 \
    --cipher aes-xts-plain64 \
    --key-size 512 \
    --hash sha256 \
    --pbkdf argon2id \
    --pbkdf-memory 1048576 \
    --iter-time 2000 \
    /dev/nvme0n1p2
Parameter explanations:
  • --cipher aes-xts-plain64: Industry-standard cipher with sector-level encryption
  • --key-size 512: XTS mode uses two keys; 256 bits each, totaling 512
  • --pbkdf argon2id: Memory-hard KDF; resistance to ASIC/GPU attacks
  • --pbkdf-memory 1048576: 1GB memory cost for Argon2
  • --iter-time 2000: 2-second unlock target (balances security/usability)

You'll be prompted for a passphrase. Use a strong, memorable passphrase—this is your primary decryption key. Consider using a passphrase manager or Diceware methodology.

2.2 Open the Encrypted Device

# Open the LUKS container, mapping to /dev/mapper/crypt_root
sudo cryptsetup open /dev/nvme0n1p2 crypt_root

# Verify it's mapped
ls -la /dev/mapper/crypt_root
sudo cryptsetup status crypt_root

The crypt_root name is arbitrary—it's used for the device mapper path. You could use any identifier, but crypt_root is conventional and clear.

2.3 Backup LUKS Header (Critical Step)

# Create backup directory
mkdir -p ~/luks-backups

# Backup header to external storage (USB, network, etc.)
sudo cryptsetup luksHeaderBackup /dev/nvme0n1p2 \
    --backup-file ~/luks-backups/luks-header-$(date +%Y%m%d).backup

# Also backup to external USB if available
# sudo cp ~/luks-backups/luks-header-*.backup /media/usb/
Critical: A corrupted LUKS header makes data unrecoverable without backups. Store this file offline, preferably in multiple locations.

Phase 3: BTRFS Setup and Subvolume Layout

BTRFS (B-tree filesystem) provides COW (Copy-on-Write), checksums, compression, and subvolumes. Unlike LVM snapshots, BTRFS snapshots are instantaneous and space-efficient—only changed blocks consume additional space.

3.1 Create BTRFS Filesystem

# Install btrfs-progs
sudo apt install -y btrfs-progs

# Create filesystem with recommended options
sudo mkfs.btrfs \
    --label debian_root \
    --features extref,skinny-metadata,quota \
    /dev/mapper/crypt_root

Parameter explanations:

  • --label: Human-readable filesystem label
  • extref: Extended inode references for better performance
  • skinny-metadata: Reduced metadata size
  • quota: Quota support for snapshot size tracking

3.2 Create Mount Point and Mount

# Create temporary mount point for initial setup
sudo mkdir -p /mnt/debian

# Mount the BTRFS root (this is the filesystem root, not the @ subvolume)
sudo mount /dev/mapper/crypt_root /mnt/debian

# Verify
sudo btrfs filesystem show /mnt/debian

3.3 Create Subvolumes

The @ naming convention comes from Ubuntu/Debian—it's the convention used by grub-btrfs and Timeshift. We'll create a production-ready layout:

# Create subvolumes
cd /mnt/debian

# System root - will become /
sudo btrfs subvolume create @

# User home directories - separate snapshot policy
sudo btrfs subvolume create @home

# Snapshot storage - excluded from @ snapshots
sudo btrfs subvolume create @snapshots

# Log files - change frequently, isolate from OS snapshots
sudo btrfs subvolume create @var_log

# Package cache - can be regenerated, isolate from backups
sudo btrfs subvolume create @var_cache

# Container/Podman storage
sudo btrfs subvolume create @var_lib_containers

# Verify
cd /mnt/debian && sudo btrfs subvolume list .

Expected output:

ID 256 gen 5 top level 5 path @
ID 257 gen 6 top level 5 path @home
ID 258 gen 7 top level 5 path @snapshots
ID 259 gen 8 top level 5 path @var_log
ID 260 gen 9 top level 5 path @var_cache
ID 261 gen 10 top level 5 path @var_lib_containers

3.4 Mount Subvolumes

Unmount the raw filesystem and mount subvolumes individually:

# Unmount root, then mount subvolumes properly
sudo umount /mnt/debian

# Remount with subvolumes
sudo mount -o subvol=@,compress=zstd:3,noatime,space_cache=v2 \
    /dev/mapper/crypt_root /mnt/debian

# Create mount points
sudo mkdir -p /mnt/debian/{boot/efi,home,.snapshots,var/log,var/cache,var/lib/containers}

# Mount additional subvolumes
sudo mount -o subvol=@home,compress=zstd:3,noatime \
    /dev/mapper/crypt_root /mnt/debian/home

sudo mount -o subvol=@snapshots,noatime \
    /dev/mapper/crypt_root /mnt/debian/.snapshots

sudo mount -o subvol=@var_log,compress=zstd:1,noatime \
    /dev/mapper/crypt_root /mnt/debian/var/log

sudo mount -o subvol=@var_cache,compress=zstd:1,noatime,nodiratime \
    /dev/mapper/crypt_root /mnt/debian/var/cache

sudo mount -o subvol=@var_lib_containers,compress=zstd:3,noatime \
    /dev/mapper/crypt_root /mnt/debian/var/lib/containers

# Mount ESP
sudo mount /dev/nvme0n1p1 /mnt/debian/boot/efi

Mount option explanations:

  • subvol=@: Mounts the subvolume instead of the filesystem root
  • compress=zstd:3: Transparent Zstd compression, level 3 (good speed/ratio)
  • noatime: Don't update access times (reduces COW churn)
  • space_cache=v2: Improved free space caching
  • nodiratime: For cache directories, don't update directory times

3.5 Verify Mount Structure

# Display mount hierarchy
findmnt /mnt/debian

# Or use df for disk usage
df -hT /mnt/debian /mnt/debian/*

Phase 4: debootstrap Base System

debootstrap creates a minimal Debian system in a target directory. Unlike the graphical installer, it gives complete control over package selection and configuration.

4.1 Install debootstrap

# Install debootstrap and essential tools
sudo apt install -y debootstrap binutils gdisk arch-install-scripts

4.2 Run debootstrap

Debian 13 is currently trixie (testing). For the latest stable, substitute stable or the release codename:

# Install base system (using trixie for Debian 13)
sudo debootstrap \
    --variant=minbase \
    --include=linux-image-amd64,locales,initramfs-tools,cryptsetup,btrfs-progs,sudo,vim,nano,curl,wget \
    trixie \
    /mnt/debian \
    http://deb.debian.org/debian/ 2>&1 | tee ~/debootstrap.log

This downloads and installs the base system. Output is tee'd to a log file for troubleshooting.

4.3 Bind Mount System Directories

The chroot needs access to runtime information from the host:

# Bind mount essential filesystems
sudo mount --bind /dev /mnt/debian/dev
sudo mount --bind /dev/pts /mnt/debian/dev/pts
sudo mount --bind /proc /mnt/debian/proc
sudo mount --bind /sys /mnt/debian/sys
sudo mount --bind /run /mnt/debian/run

# For networking inside chroot
sudo cp /etc/resolv.conf /mnt/debian/etc/resolv.conf

4.4 Enter Chroot Environment

# Enter chroot with full environment
sudo chroot /mnt/debian /bin/bash

# Welcome to your new Debian system!
ls /
whoami  # Should show 'root'
From this point until Phase 6, all commands execute inside the chroot unless explicitly stated otherwise.

Phase 5: System Configuration Inside Chroot

Now we configure the base system: networking, authentication, crypto integration, and package management.

5.1 Configure Filesystem Table (fstab)

Use UUIDs for device identification—resistant to device name changes:

# Get UUIDs
blkid /dev/nvme0n1p1
blkid /dev/nvme0n1p2

# Create fstab
cat > /etc/fstab << 'EOF'
# /etc/fstab - filesystem table
#                                                                                 

# Encrypted BTRFS root
UUID=PLACEHOLDER_CRYPT_ROOT   /               btrfs   subvol=@,compress=zstd:3,noatime,space_cache=v2     0      0

# Subvolumes
UUID=PLACEHOLDER_CRYPT_ROOT   /home           btrfs   subvol=@home,compress=zstd:3,noatime                  0      0
UUID=PLACEHOLDER_CRYPT_ROOT   /.snapshots     btrfs   subvol=@snapshots,noatime                             0      0
UUID=PLACEHOLDER_CRYPT_ROOT   /var/log        btrfs   subvol=@var_log,compress=zstd:1,noatime               0      0
UUID=PLACEHOLDER_CRYPT_ROOT   /var/cache      btrfs   subvol=@var_cache,compress=zstd:1,noatime,nodiratime    0      0
UUID=PLACEHOLDER_CRYPT_ROOT   /var/lib/containers  btrfs   subvol=@var_lib_containers,compress=zstd:3,noatime  0      0

# EFI System Partition
UUID=PLACEHOLDER_ESP          /boot/efi       vfat    umask=0077,codepage=437,iocharset=ascii,shortname=winnt 0      1

# Tmpfs for performance
tmpfs                                      /tmp            tmpfs   defaults,nosuid,nodev,size=50%                      0      0
EOF

# Now substitute UUIDs
crypt_uuid=$(blkid -s UUID -o value /dev/nvme0n1p2)
esp_uuid=$(blkid -s UUID -o value /dev/nvme0n1p1)

sed -i "s/PLACEHOLDER_CRYPT_ROOT/$crypt_uuid/g" /etc/fstab
sed -i "s/PLACEHOLDER_ESP/$esp_uuid/g" /etc/fstab

# Verify
cat /etc/fstab

5.2 Configure crypttab for Auto-Unlock

# Create crypttab with the same mapper name
echo "crypt_root UUID=$crypt_uuid none luks,discard" > /etc/crypttab

# Verify
cat /etc/crypttab
Note: discard enables TRIM on SSDs, which can weaken security against sophisticated attacks but is required for SSD longevity. Remove if using a security-focused threat model.

5.6 Configure Initramfs for Encrypted Boot

The initramfs must unlock the LUKS container before mounting root:

# Ensure cryptsetup is included in initramfs
apt install -y cryptsetup-initramfs

# Update initramfs

# Regenerate initramfs for all kernels

# Verify the generated image contains crypto modules

5.7 Set Root Password

# Set secure root password

Use a strong, unique password. This is for emergency recovery only.

5.8 Create Regular User

# Create user with sudo privileges
useradd -m -s /bin/bash -G sudo,audio,video,cdrom,floppy,adm,users,netdev stefan
passwd stefan

# Configure sudo for no-password (optional, security trade-off)
echo 'stefan ALL=(ALL) NOPASSWD:ALL' > /etc/sudoers.d/stefan
chmod 440 /etc/sudoers.d/stefan

Phase 6: Bootloader and grub-btrfs

Exit the chroot and configure the bootloader from the live environment.

# Exit chroot
exit

# Return to live environment prompt
# Verify you're outside chroot (prompt changes)

6.1 Install GRUB Bootloader

# Install GRUB for UEFI
sudo apt install -y grub-efi-amd64 grub-efi-amd64-signed shim-signed

# Install GRUB to target system
sudo grub-install \
    --target=x86_64-efi \
    --efi-directory=/mnt/debian/boot/efi \
    --bootloader-id=debian \
    --root-directory=/mnt/debian \
    --removable
--removable flag: Creates /EFI/BOOT/BOOTX64.EFI, the fallback boot path. Useful for removable media and ensures boot even if NVRAM entries are lost.

6.2 Build grub-btrfs from Source

grub-btrfs adds boot menu entries for BTRFS snapshots. This is the feature that makes snapshots truly useful.

# Install dependencies
sudo apt install -y \
    git \
    make \
    gawk \
    btrfs-progs \
    grub-common

# Clone repository
cd /tmp
git clone https://github.com/Antynea/grub-btrfs.git
cd grub-btrfs

# Checkout latest stable (v4.12 as of 2026)
git checkout v4.12

# Review the Makefile
less Makefile

6.3 Configure and Install grub-btrfs

# Review configuration defaults
cat config

Edit the config before installation. Key settings to verify:

# Create config override
cat > /tmp/grub-btrfs/config << 'EOF'
# grub-btrfs configuration

# Enable snapshots in boot menu
GRUB_BTRFS_ENABLE="true"

# Snapshot submenu name
GRUB_BTRFS_SUBMENUNAME="Snapshots"

# Show snapshot name in menu
GRUB_BTRFS_SHOW_SNAPSHOTS_FOUND="true"

# Number of snapshots to show
GRUB_BTRFS_LIMIT="50"

# Subvolume where snapshots are stored
GRUB_BTRFS_SUBVOLUME="/.snapshots"

# Name of the snapshot directory
GRUB_BTRFS_SNAPSHOT_SUBMENU="Snapshot"

# Add version tag to snapshot name
GRUB_BTRFS_SHOW_VERSION="true"

# Date format
GRUB_BTRFS_DATE_FORMAT="%Y-%m-%d %H:%M"

# Custom kernel parameters for snapshots (important!)
GRUB_BTRFS_SNAPSHOT_KERNEL_PARAMETERS="rootflags=subvol=@snapshots/@snapshot_name"

# Disable fallback (we'll use initramfs)
GRUB_BTRFS_DISABLE_FALLBACK_SNAPSHOT="true"
EOF

Now install:

# Install to target system
cd /tmp/grub-btrfs
sudo make install DESTDIR=/mnt/debian

# Verify installation
ls -la /mnt/debian/etc/grub.d/ | grep btrfs

6.4 Configure GRUB for LUKS and BTRFS

# Edit GRUB default configuration
sudo nano /mnt/debian/etc/default/grub

Key modifications:

# GRUB config - ensure these settings
GRUB_DEFAULT=0
GRUB_TIMEOUT=10
GRUB_DISTRIBUTOR="Debian"

# Critical: enable cryptodisk for LUKS
GRUB_ENABLE_CRYPTODISK=y

# Enable BTRFS support
GRUB_PRELOAD_MODULES="part_gpt part_msdos cryptodisk luks btrfs"

# Kernel command line - UUID and crypt_root
GRUB_CMDLINE_LINUX="cryptdevice=UUID=YOUR_CRYPT_UUID:crypt_root root=/dev/mapper/crypt_root"

# Quiet boot with encryption prompts
GRUB_CMDLINE_LINUX_DEFAULT="quiet boot=live persistent"

# Show menu (not hidden)
GRUB_TIMEOUT_STYLE=menu

# Use GFXterm for better display
GRUB_TERMINAL=console

# Don't probe other OS (pure Debian)
GRUB_DISABLE_OS_PROBER=true
Replace YOUR_CRYPT_UUID with the actual UUID from blkid /dev/nvme0n1p2

6.5 Generate GRUB Configuration

# Re-enter chroot to run update-grub
sudo chroot /mnt/debian /bin/bash

# Generate GRUB config
update-grub

Phase 7: Install Timeshift for Snapshots

Timeshift provides a user-friendly interface for creating and restoring snapshots, plus scheduled backup policies.

7.1 Install Timeshift

# Re-enter chroot
sudo chroot /mnt/debian /bin/bash

# Add Timeshift repository (not in Debian default yet for latest version)
apt install -y software-properties-common
deb [arch=amd64] http://ppa.launchpad.net/teejee2008/timeshift/ubuntu jammy main

# Or install from Debian repos (older but stable version)
apt install -y timeshift

# Configure Timeshift for BTRFS
mkdir -p /etc/timeshift

cat > /etc/timeshift/timeshift.json << 'EOF'
{
    "backup_device_uuid": "YOUR_CRYPT_UUID_HERE",
    "parent_device_uuid": "",
    "do_first_run": false,
    "first_run": false,
    "backup_type": "btrfs",
    "snapshot_type": "btrfs",
    "snapshot_levels": [1, 2, 3, 4, 5],
    "snapshot_level_annotations": ["Hourly", "Daily", "Weekly", "Monthly", "Boot"],
    "snapshots_to_keep": [10, 7, 4, 6, 3],
    "schedule_monthly": false,
    "schedule_weekly": true,
    "schedule_daily": true,
    "schedule_hourly": true,
    "schedule_boot": true,
    "include_btrfs_home": false,
    "exclude": [
        "/home/stefan/.cache",
        "/home/stefan/.var",
        "/home/stefan/Downloads",
        "/tmp/**"
    ],
    "btrfs_use_qgroups": false
}
EOF

7.2 Configure Timeshift Service

# Enable Timeshift's systemd timers
systemctl enable timeshift.timer
systemctl enable timeshift-boot.timer

# Create initial snapshot
# (Run this after first boot)
# timeshift --create --comments "First Boot - Clean Install"
Timeshift tip: Timeshift only snapshots the @ subvolume by default. This is why we separated @home—it won't rollback user data accidentally.

Phase 11: Rollback Procedure

When you need to recover:

11.1 Boot from Snapshot

  1. Reboot system
  2. Hold Shift during boot to enter GRUB menu
  3. Select "Snapshots" submenu
  4. Choose desired snapshot (shows date/time in name)
  5. Boot that snapshot (read-only mode)

11.2 Perform Rollback

Booted from snapshot, open a terminal:

# Mount root filesystem
sudo mount /dev/mapper/crypt_root /mnt

# Navigate to snapshots directory
cd /mnt/@snapshots

# List available snapshots
ls -la

# Rollback: rename current @ to @bad, rename snapshot @ to @
sudo btrfs subvolume delete @/  # CAREFUL: This DELETES current root

# Or safer: rename instead of delete
sudo mv /mnt/@ /mnt/@bad_$(date +%Y%m%d_%H%M)
sudo btrfs subvolume snapshot /mnt/@snapshots/YYYY-MM-DD_HH-MM-SS /@

# Unmount
sudo umount /mnt

# Reboot to rolled-back system
sudo reboot

Maintenance and Best Practices

Snapshot Hygiene

# List snapshot disk usage
sudo btrfs filesystem du -s /.snapshots/*

# Delete old snapshots
sudo timeshift --delete --snapshot '2026-01-15_12-00-00'

# Balance filesystem (reclaim space)
sudo btrfs filesystem defragment -r /

# Scrub for data integrity
sudo btrfs scrub start /
sudo btrfs scrub status /

Backup Strategy

Snapshots are not backups—they protect against human error, not hardware failure. Supplement with:

  • rsync to external storage for @home
  • borg or restic for encrypted cloud backups
  • LUKS header backup as completed in Phase 2.3

LUKS Key Management

# Add a YubiKey/TPM as additional key slot (optional)
# cryptsetup luksAddKey /dev/nvme0n1p2 --key-slot 1

# Check key slots
sudo cryptsetup luksDump /dev/nvme0n1p2 | grep -A2 "Keyslots"

# Remove a key slot
# sudo cryptsetup luksRemoveKey /dev/nvme0n1p2 --key-slot 1

Troubleshooting

Issue Solution
grub-install: error: cannot find EFI directory ESP not mounted on /boot/efi. Check findmnt /boot/efi
cryptsetup: warning: could not determine mountpoint Fstab entry malformed. Verify blkid UUIDs match
Boot drops to initramfs prompt Root device not found. Check etc/crypttab has correct UUID
grub-btrfs menu empty No snapshots exist. Create one with timeshift --create
Wayland session not available Check systemctl status gdm and /var/log/gdm/
Timeshift fails to create snapshot Ensure @snapshots subvolume exists and is mounted
BTRFS out of space Run btrfs filesystem sync then balance

Performance Considerations

Compared to ext4 + LVM:

  • Boot time: +2-3 seconds (LUKS unlock overhead)
  • Write throughput: -5-10% (Zstd compression overhead)
  • Snapshot creation: Instant vs minutes for LVM
  • Snapshot space: COW-only changed blocks vs full copy
  • SSD longevity: Improved (no journal writes, compression reduces writes)

  • ✅ One-click rollback via Timeshift

This architecture mirrors enterprise Linux distributions while maintaining Debian's stability and flexibility. The snapshot/rollback capability transforms system administration from risk management to experimentation—you can test kernel upgrades, major software changes, and configuration experiments with confidence that recovery is seconds away.

The LUKS2+BTRFS combination is battle-tested in Fedora, OpenSUSE, and RHEL. By implementing it on Debian, you get the best of both worlds: enterprise-grade storage management on the distribution known for its stability.
