Debian with LUKS2, BTRFS & Snapper: Production Installation Guide

Complete terminal-based walkthrough: From Debian net-installer to production-ready GNOME system with encrypted BTRFS, automatic snapshots, and bootable recovery.

Goal: Install Debian testing with LUKS2 encryption, BTRFS with Kali "Unkaputtbar" subvolumes (including autodefrag), separate unencrypted /boot, Snapper for automatic APT snapshots, and full GNOME desktop.

Architecture

Three-partition layout with unencrypted /boot for reliability and encrypted BTRFS root with Kali-style subvolumes:

nvme0n1p1  1GB   FAT32   /boot/efi     (ESP - UEFI bootloader, unencrypted)
nvme0n1p2  2GB   ext4    /boot         (kernels & initramfs - unencrypted)
nvme0n1p3  Rest  LUKS2   crypt_root    (encrypted BTRFS container)
           └── @           (root filesystem)
           ├── @home       (/home - user data)
           ├── @root       (/root - root home)
           ├── @srv        (/srv - site data)
           ├── @tmp        (/tmp - temp files)
           ├── @usr_local  (/usr/local - local admin)
           ├── @var_log    (/var/log - system logs)
           └── @.snapshots (snapper snapshots)

Understanding the Mount Options

Each subvolume is mounted with specific BTRFS options for optimal performance:

  • noatime: Disables atime updates for less disk I/O
  • compress=zstd:1: Enables zstd compression level 1 (lightweight)
  • autodefrag: Automatically defragments files in the background (added to data subvolumes)
  • discard=async: Async TRIM for SSDs
  • nodiratime: Disables directory atime (used on @tmp where we don't need access times at all)

Note: autodefrag is not added to @tmp (temporary data doesn't need defrag) or @.snapshots (snapshots are immutable).

Phase 1: Boot Installer & Enter Terminal

Why: We need access to the Debian net-installer's live environment to manually partition and debootstrap.

  1. Boot from Debian net-install ISO
  2. At the boot menu, press Tab to edit parameters if needed
  3. Press Enter to boot

When the installer loads (language/location screens), switch to TTY2:

# Press: Ctrl + Alt + F2
# Then press Enter to activate the terminal

Phase 2: Partition the Disk

Why 3 partitions? Separating /boot (unencrypted ext4) from the encrypted root allows GRUB to load the kernel without needing LUKS2 support in the bootloader. The ESP holds GRUB's EFI executable.

# Identify your target disk (look for nvme, sda, vda, etc.)
lsblk -dpno NAME,SIZE,MODEL | grep -E 'nvme|sd|vd'

# Set disk variable (adjust as needed)
DISK="/dev/nvme0n1"

# Create GPT partition table and 3 partitions:
# p1: 1GB ESP (ef00 = EFI System Partition)
# p2: 2GB Linux filesystem (8300 = Linux filesystem)
# p3: Remainder for LUKS2 encrypted root
sgdisk --zap-all "$DISK"
sgdisk -n "1:0:+1G" -t "1:ef00" -c "1:ESP" "$DISK"
sgdisk -n "2:0:+2G" -t "2:8300" -c "2:BOOT" "$DISK"
sgdisk -n "3:0:0" -t "3:8300" -c "3:CRYPT" "$DISK"
lsblk "$DISK"

Phase 3: Format Partitions

What each step does:

  • ESP: FAT32 filesystem required by UEFI firmware
  • /boot: ext4 for kernel storage; tune2fs -m 0 frees reserved blocks (not needed on boot)
  • LUKS2: Full disk encryption using AES-XTS with Argon2 key derivation
  • BTRFS: Modern filesystem with snapshots, compression, and subvolumes
# Format ESP with FAT32 label
mkfs.fat -F32 -n ESP "${DISK}p1"

# Format /boot with ext4, remove reserved blocks
mkfs.ext4 -L BOOT "${DISK}p2"
tune2fs -m 0 "${DISK}p2"

# Setup LUKS2 on root partition
# You'll be prompted to create a passphrase (256-bit encryption)
cryptsetup luksFormat --type luks2 "${DISK}p3"

# Open the encrypted container
# This creates /dev/mapper/crypt_root as the decrypted block device
cryptsetup open "${DISK}p3" crypt_root

# Create BTRFS filesystem on the opened LUKS container
mkfs.btrfs -L DEBIAN /dev/mapper/crypt_root

Phase 4: Create BTRFS Subvolumes

Why subvolumes? Separating directories into subvolumes allows independent snapshotting. The Kali "Unkaputtbar" layout isolates user data, logs, and temp files from system changes.

# Mount the raw BTRFS root to create subvolumes
mount /dev/mapper/crypt_root /mnt

# Create Kali "Unkaputtbar" subvolume layout
btrfs subvolume create /mnt/@           # Root filesystem
btrfs subvolume create /mnt/@home       # User home directories
btrfs subvolume create /mnt/@root       # Root user's home
btrfs subvolume create /mnt/@srv        # Service data
btrfs subvolume create /mnt/@tmp        # Temporary files (no autodefrag)
btrfs subvolume create /mnt/@usr_local  # Locally installed software
btrfs subvolume create /mnt/@var_log    # System logs
btrfs subvolume create /mnt/@.snapshots # Snapper snapshot storage

# Unmount raw root - we'll mount subvolumes individually
umount /mnt

Phase 5: Mount All Filesystems

The mount strategy: Each subvolume gets mounted with optimized options. Data subvolumes (@, @home, @root, @srv, @usr_local) use compress=zstd:1,autodefrag. The logs subvolume skips compression. Temp gets noatime,nodiratime for minimal updates.

# Mount root subvolume with compression and autodefrag (keeps system defragmented)
mount -o noatime,compress=zstd:1,autodefrag,subvol=@ /dev/mapper/crypt_root /mnt
mkdir -p /mnt/{boot,home,root,srv,tmp,usr/local,var/log,.snapshots}

# Mount data subvolumes with compression + autodefrag to prevent fragmentation
# autodefrag runs in the background and keeps frequently modified files contiguous
mount -o noatime,compress=zstd:1,autodefrag,subvol=@home /dev/mapper/crypt_root /mnt/home
mount -o noatime,compress=zstd:1,autodefrag,subvol=@root /dev/mapper/crypt_root /mnt/root
mount -o noatime,compress=zstd:1,autodefrag,subvol=@srv /dev/mapper/crypt_root /mnt/srv

# tmp: no compression, no access times at all (not needed for temp files)
mount -o noatime,nodiratime,subvol=@tmp /dev/mapper/crypt_root /mnt/tmp

# /usr/local: autodefrag helps with software compilation
mount -o noatime,compress=zstd:1,autodefrag,subvol=@usr_local /dev/mapper/crypt_root /mnt/usr/local

# /var/log: autodefrag for rotating log writes (no compression = better for append-only)
mount -o noatime,autodefrag,subvol=@var_log /dev/mapper/crypt_root /mnt/var/log

# snapshots: no autodefrag (immutable), no compression (already compressed by zstd)
mount -o noatime,subvol=@.snapshots /dev/mapper/crypt_root /mnt/.snapshots

# Mount unencrypted partitions
mount "${DISK}p2" /mnt/boot
mkdir -p /mnt/boot/efi
mount "${DISK}p1" /mnt/boot/efi

# Verify mount layout
lsblk "$DISK"
df -hT | grep /mnt

Phase 6: Bootstrap Debian with debootstrap

What is debootstrap? It downloads and extracts Debian packages to create a minimal base system. We use "testing" to get the latest stable-but-rolling packages.

# Install debootstrap if not present (in Debian installer, may already be there)
debootstrap --version || apt update && apt install -y debootstrap

# Bootstrap testing base system into /mnt
# This downloads ~300MB of packages and creates the root filesystem
debootstrap testing /mnt http://deb.debian.org/debian

# Mount pseudo-filesystems for chroot
# --rbind propagates mount events; --make-rslave prevents propagation back
cd /mnt
mount --make-rslave --rbind /dev dev
mount --make-rslave --rbind /sys sys
mount --make-rslave --rbind /proc proc
mount --make-rslave --rbind /run run

Phase 7: Enter Chroot & Configure Base System

What's a chroot? It changes the apparent root directory, allowing us to configure the installed system as if we booted into it.

# Enter chroot environment
chroot /mnt /bin/bash

# Set hostname (replace "your-hostname" with actual name)
echo "your-hostname" > /etc/hostname

# Configure hosts file for local name resolution
cat > /etc/hosts << 'EOF'
127.0.0.1   localhost
127.0.1.1   your-hostname
::1         localhost ip6-localhost ip6-loopback
ff02::1     ip6-allnodes
ff02::2     ip6-allrouters
EOF

# Set root password (interactively prompted)
passwd

# Create user with sudo access (replace "your-username")
useradd -m -s /bin/bash -G sudo your-username
passwd your-username

# Configure APT for Debian testing with contrib/non-free firmware
cat > /etc/apt/sources.list << 'EOF'
deb http://deb.debian.org/debian testing main contrib non-free non-free-firmware
deb http://deb.debian.org/debian testing-updates main contrib non-free non-free-firmware
deb http://security.debian.org/debian-security testing-security main contrib non-free non-free-firmware
EOF

# Refresh package lists
apt update

Phase 8: Install Kernel & Essential Packages

Package groups:

  • Kernel: linux-image-amd64 + headers for DKMS
  • Firmware: Required for WiFi, GPU, etc.
  • System: networking, crypto tools, bootloader essentials
  • Desktop: Full GNOME via task-gnome-desktop meta-package
# Install kernel and firmware (large download ~500MB)
apt install -y linux-headers-amd64 linux-image-amd64 firmware-linux \
    firmware-linux-nonfree firmware-misc-nonfree

# Install core system packages
apt install -y network-manager wireless-tools wpasupplicant \
    cryptsetup-initramfs btrfs-progs grub-efi-amd64 sudo

# Install full GNOME desktop environment
apt install -y task-gnome-desktop gdm3

Phase 9: Configure fstab

What is fstab? It defines how filesystems mount at boot. We generate it with proper UUIDs (which remain constant even if device names change).

# Get UUIDs for persistent mounting
crypt_uuid=$(blkid -s UUID -o value "${DISK}p3")
boot_uuid=$(blkid -s UUID -o value "${DISK}p2")
esp_uuid=$(blkid -s UUID -o value "${DISK}p1")

# Generate fstab with autodefrag on data subvolumes
cat > /etc/fstab << EOF
# /etc/fstab - LUKS2 BTRFS with autodefrag optimization
#                                                                      

# Root filesystem with autodefrag (keeps OS files defragmented)
UUID=${crypt_uuid}  /              btrfs   noatime,compress=zstd:1,autodefrag,subvol=@,discard=async                          0 1

# User data with autodefrag (documents, downloads benefit from sequential access)
UUID=${crypt_uuid}  /home          btrfs   noatime,compress=zstd:1,autodefrag,subvol=@home,discard=async                    0 2

# Root user home (same optimization)
UUID=${crypt_uuid}  /root          btrfs   noatime,compress=zstd:1,autodefrag,subvol=@root,discard=async                    0 2

# Service data with autodefrag
UUID=${crypt_uuid}  /srv           btrfs   noatime,compress=zstd:1,autodefrag,subvol=@srv,discard=async                     0 2

# Temp - no autodefrag (temporary data), no compression
UUID=${crypt_uuid}  /tmp           btrfs   noatime,nodiratime,subvol=@tmp,discard=async                            0 2

# Local software - autodefrag helps with compiled/installed files
UUID=${crypt_uuid}  /usr/local     btrfs   noatime,compress=zstd:1,autodefrag,subvol=@usr_local,discard=async               0 2

# Logs - autodefrag for better append performance, no compression (already small)
UUID=${crypt_uuid}  /var/log       btrfs   noatime,autodefrag,subvol=@var_log,discard=async                               0 2

# Snapshots - no autodefrag (snapshots are immutable), no compression
UUID=${crypt_uuid}  /.snapshots    btrfs   noatime,subvol=@.snapshots,discard=async                                      0 2

# /boot (unencrypted ext4) - standard mount
UUID=${boot_uuid}   /boot          ext4    defaults,noatime                                                        0 2

# EFI System Partition - required for UEFI boot
UUID=${esp_uuid}    /boot/efi      vfat    umask=0077,noatime                                                    0 1
EOF

# Verify fstab
cat /etc/fstab

Phase 10: Configure Crypttab & Initramfs

crypttab tells the initramfs which encrypted devices to unlock at boot. initramfs is the early userspace loaded by GRUB to unlock LUKS before mounting root.

# Configure crypttab for boot-time unlocking
# "none" means prompt for passphrase (LUKS header contains keyslots)
echo "crypt_root UUID=${crypt_uuid} none luks,discard" > /etc/crypttab
cat /etc/crypttab

# Install CPU microcode for early CPU bug fixes (improves boot stability)
apt install -y amd64-microcode || apt install -y intel-microcode

# Generate initramfs images for all installed kernels
# This embeds crypttab, cryptsetup, and btrfs tools into the initramfs
update-initramfs -u -k all

Phase 11: Install GRUB Bootloader

GRUB installation: The bootloader is installed to the ESP, configured to unlock LUKS via cryptdevice= kernel parameter, and generates a boot menu.

# Install GRUB EFI binary to ESP
grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=debian --recheck

# Configure GRUB defaults
cat > /etc/default/grub << 'EOF'
GRUB_DEFAULT=0
GRUB_TIMEOUT=5
GRUB_DISTRIBUTOR='Debian'
GRUB_CMDLINE_LINUX_DEFAULT="quiet"
GRUB_CMDLINE_LINUX="cryptdevice=UUID=PLACEHOLDER:crypt_root"
GRUB_TERMINAL=console
GRUB_GFXMODE=1024x768
EOF

# Replace placeholder with actual LUKS UUID
sed -i "s/PLACEHOLDER/${crypt_uuid}/g" /etc/default/grub

# Generate grub.cfg with detected kernels
cat /etc/default/grub
update-grub

Phase 12: Install Snapper & Configure

Snapper workflow: APT pre-hooks create ">before" snapshots, post-hooks create "after" snapshots. You can rollback to any snapshot and boot from them via GRUB.

# Install Snapper and GUI frontend
apt install -y snapper snapper-gui

# Create snapper config for the root subvolume
snapper --config root create-config /

# Configure snapper retention policy
cat > /etc/snapper/configs/root << 'EOF'
# snapper configuration for root filesystem
SUBVOLUME="/"
FSTYPE="btrfs"
QGROUP=""
SPACE_LIMIT="0.5"
FREE_LIMIT="0.2"

# Disable timeline (we use APT hooks, not cron)
TIMELINE_CREATE="no"
TIMELINE_CLEANUP="no"

# Keep last 10 numbered snapshots (manual/APT), clean older
NUMBER_CLEANUP="yes"
NUMBER_MIN_AGE="1800"
NUMBER_LIMIT="10"
EMPTY_PRE_POST_CLEANUP="yes"
EOF

# Install python-dbus for snapper integration
apt install -y python3-dbus

# Create APT pre-hook: snapshot BEFORE any install/remove/upgrade
cat > /etc/apt/apt.conf.d/80-snapper-pre << 'EOF'
// Create snapshot before APT operations
DPkg::Pre-Invoke {"if which snapper >/dev/null 2>&1; then snapper --config root create --description \"apt-pre-$(date +%Y%m%d-%H%M%S)\" --cleanup-algorithm number 2>/dev/null || true; fi"; };
EOF

# Create APT post-hook: snapshot AFTER any install/remove/upgrade completes
cat > /etc/apt/apt.conf.d/80-snapper-post << 'EOF'
// Create snapshot after APT operations
DPkg::Post-Invoke {"if which snapper >/dev/null 2>&1; then snapper --config root create --description \"apt-post-$(date +%Y%m%d-%H%M%S)\" --cleanup-algorithm number 2>/dev/null || true; fi"; };
EOF

Phase 13: Install grub-btrfs (Bootable Snapshots)

grub-btrfs automatically finds BTRFS snapshots and adds them to the GRUB menu. You can boot into any snapshot for recovery (they're read-write, so you can test).

# Install build dependencies
apt install -y git build-essential gettext

# Clone grub-btrfs from GitHub
cd /tmp
git clone https://github.com/Antynea/grub-btrfs.git
cd grub-btrfs

# Install grub-btrfs (adds snapshot submenu to GRUB)
make install

# Update GRUB to detect any existing snapshots
update-grub

Phase 14: Exit Chroot, Unmount, & Reboot

Clean shutdown: Unmount in reverse order, close LUKS, reboot. On first boot, you'll enter your passphrase to unlock the root partition.

# Exit the chroot (return to live environment)
exit

# Unmount pseudo-filesystems (important before unmounting root)
umount -R /mnt/dev
umount -R /mnt/sys
umount -R /mnt/proc
umount -R /mnt/run

# Unmount boot partitions
umount /mnt/boot/efi
umount /mnt/boot

# Unmount all BTRFS subvolumes
umount -R /mnt

# Close LUKS container
cryptsetup close crypt_root

# Reboot into the new system
reboot

First Boot: LUKS Unlock

On boot:

  1. GRUB menu appears (timeout 5 seconds)
  2. Kernel loads, initramfs prompts: crypt_root: please enter passphrase:
  3. Enter your LUKS passphrase to unlock the root partition
  4. System continues booting to GDM login screen

Working with Snapshots

Viewing and managing snapshots:

# List all snapshots (pre/post/manual)
snapper --config root list

# Create manual snapshot before big changes
snapper --config root create --description "Before kernel upgrade"

# Delete specific snapshot by number
snapper --config root delete 3

# Compare two snapshots (shows file changes)
snapper --config root diff 1..2

# Rollback to snapshot (reverts @ subvolume, requires reboot)
snapper --config root rollback 3

Boot from Snapshot (Recovery Workflow)

Why boot from snapshot? If an upgrade breaks your system, boot into a pre-upgrade snapshot, verify it works, then make it permanent with snapper rollback.

  1. Reboot and hold Shift during boot to show GRUB menu
  2. Select Debian Snapshots submenu
  3. Choose any snapshot (they boot read-write, not read-only)
  4. Log in and test the system
  5. If satisfied: sudo snapper rollback <number> and reboot

Verification Commands

# List all BTRFS subvolumes
btrfs subvolume list / | sort

# Check filesystem compression usage
btrfs filesystem df /

# Check LUKS container status
cryptsetup status crypt_root

# View snapper configuration
snapper get-config

# Check GRUB environment
grub-editenv list

# Verify mount options (should show autodefrag)
mount | grep btrfs

About autodefrag

The autodefrag mount option automatically defragments files in the background:

  • When a file is written and becomes fragmented, BTRFS queues it for defrag
  • Runs during idle I/O periods (low priority)
  • Reduces fragmentation on frequently modified files (databases, VM images, browser profiles)
  • Not applied to @tmp (temporary) or @.snapshots (immutable data)

See BTRFS documentation for details.
