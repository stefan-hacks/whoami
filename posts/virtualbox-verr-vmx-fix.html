To resolve the VERR_VMX_IN_VMX_ROOT_MODE error in VirtualBox on Debian Linux

The Problem

The error occurs because the KVM hypervisor is active and monopolizes hardware virtualization (VMX), preventing VirtualBox from accessing it. This commonly happens on Debian-based systems where KVM might be enabled by default or installed as a dependency.

Error Message: VERR_VMX_IN_VMX_ROOT_MODE

Solutions

Choose one of the methods below based on your needs:

Method 1: Unload KVM Modules Temporarily (Test First)

This method unloads KVM modules but they will reload after reboot:

bash 
# Unload KVM modules
sudo rmmod kvm_intel kvm_amd kvm

# Verify modules are unloaded
lsmod | grep kvm
# Should return nothing

# Launch VirtualBox VM

Method 2: Disable KVM Services

If you don't need KVM at all, stop and disable the services:

bash 
# Stop KVM services
sudo systemctl stop libvirtd
sudo systemctl stop qemu-kvm

# Disable from starting at boot
sudo systemctl disable libvirtd
sudo systemctl disable qemu-kvm

Method 3: Blacklist KVM Modules (Permanent Solution)

Add KVM modules to the blacklist so they never load:

bash 
# Create blacklist file
echo "blacklist kvm" | sudo tee /etc/modprobe.d/blacklist-kvm.conf
echo "blacklist kvm_intel" | sudo tee -a /etc/modprobe.d/blacklist-kvm.conf
echo "blacklist kvm_amd" | sudo tee -a /etc/modprobe.d/blacklist-kvm.conf

# Update initramfs
sudo update-initramfs -u

# Reboot
sudo reboot

Method 4: Remove KVM Packages

If you don't need virtualization at all:

bash 
# Remove KVM/QEMU packages
sudo apt remove --purge qemu-kvm libvirt-daemon-system virt-manager
sudo apt autoremove

# Reboot
sudo reboot

Verification

After applying any method, verify KVM is disabled:

bash 
# Check if KVM is loaded
lsmod | grep kvm

# Check KVM device
ls /dev/kvm
# Should return "No such file or directory"

# Test VirtualBox
virtualbox

Understanding the Root Cause

Both KVM and VirtualBox require direct access to hardware virtualization features (Intel VT-x or AMD-V). However, only one hypervisor can control these features at a time. When KVM loads its kernel modules, it takes exclusive control of the virtualization extensions, preventing VirtualBox from accessing them.

This is a fundamental limitation in how CPU virtualization works - the hardware can only have one hypervisor manager at a time.

When to Keep KVM

Consider keeping KVM if you:

  • Use virt-manager or qemu-kvm for VMs
  • Run containers with high performance
  • Need nested virtualization
  • Use Kubernetes with containerd

In this case, use Method 1 to temporarily unload modules when you need VirtualBox.
