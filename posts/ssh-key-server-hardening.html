SSH Key Creation, Deployment, and Server Hardening Guide

Comprehensive guide to key-based authentication, server hardening, and fail2ban implementation for Linux administrators.

Introduction

SSH key-based authentication is the gold standard for server access. This guide covers generating Ed25519 keys, deploying to multiple servers, and implementing comprehensive security hardening with fail2ban.

Security Principle: Never use password authentication. Keys are unique, revocable, and resistant to brute force attacks.

1. Generate SSH Key Pair

Create a dedicated Ed25519 key for server administration:

bash
# Create Ed25519 key (recommended)
ssh-keygen -t ed25519 -a 100 \
  -f ~/.ssh/id_ed25519_admin \
  -C "admin@$(hostname)-$(date +%Y%m%d)"

# Alternative: RSA for older systems
ssh-keygen -t rsa -b 4096 -o -a 100 \
  -f ~/.ssh/id_rsa_admin \
  -C "admin@$(hostname)-$(date +%Y%m%d)"

2. Configure SSH Client

Create a clean ~/.ssh/config to manage multiple servers:

~/.ssh/config
Host server1
    HostName 192.168.1.10
    User admin
    IdentityFile ~/.ssh/id_ed25519_admin
    Port 22

Host server2
    HostName 192.168.1.11
    User admin
    IdentityFile ~/.ssh/id_ed25519_admin

Host *
    PasswordAuthentication no
    PubkeyAuthentication yes
    UseKeychain yes
    AddKeysToAgent yes
bash
chmod 600 ~/.ssh/config

3. Deploy Keys to Servers

bash
# Add key to SSH agent
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519_admin

# Deploy to multiple servers
ssh-copy-id -i ~/.ssh/id_ed25519_admin.pub admin@server1
ssh-copy-id -i ~/.ssh/id_ed25519_admin.pub admin@server2
ssh-copy-id -i ~/.ssh/id_ed25519_admin.pub admin@server3

4. Secure SSH Server Configuration

Create a hardening script to run on each server:

secure-ssh.sh
#!/bin/bash
# SSH Hardening Script - Run on each server

BACKUP_DIR="/etc/ssh/backup_$(date +%Y%m%d)"
mkdir -p $BACKUP_DIR
cp /etc/ssh/sshd_config $BACKUP_DIR/

cat > /etc/ssh/sshd_config << 'EOF'
Port 22
Protocol 2

# Authentication (keys only)
PubkeyAuthentication yes
PasswordAuthentication no
ChallengeResponseAuthentication no
GSSAPIAuthentication no

# Security
PermitRootLogin no
PermitEmptyPasswords no
StrictModes yes
MaxAuthTries 3
MaxSessions 5
LoginGraceTime 60

# Network
X11Forwarding no
TCPKeepAlive yes
ClientAliveInterval 300
ClientAliveCountMax 2

# Crypto ( secure algorithms)
Ciphers chacha20-poly1305@openssh.com,aes256-gcm@openssh.com
MACs hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com
EOF

chmod 600 /etc/ssh/sshd_config
systemctl restart sshd
echo "✓ SSH hardening complete"

5. Test SSH Connections

test-connections.sh
#!/bin/bash
SERVERS=("server1" "server2" "server3")

echo "Testing SSH key authentication..."
for server in "${SERVERS[@]}"; do
    echo -n "Testing $server... "
    ssh -o BatchMode=yes -o ConnectTimeout=5 \
        -i ~/.ssh/id_ed25519_admin admin@$server \
        "echo '✓'" 2>/dev/null
    
    if [ $? -eq 0 ]; then
        echo "✓ Key auth working"
    else
        echo "✗ Failed"
    fi
done

6. Setup fail2ban (Recommended)

bash
sudo apt install fail2ban
sudo cp /etc/fail2ban/jail.conf /etc/fail2ban/jail.local

cat >> /etc/fail2ban/jail.local << 'EOF'
[sshd]
enabled = true
port = ssh
filter = sshd
maxretry = 3
bantime = 3600
findtime = 600
EOF

sudo systemctl restart fail2ban

Security Checklist

  • ✓ Private key permissions: chmod 600 ~/.ssh/id_ed25519_admin
  • ✓ Use strong passphrase on private key
  • ✓ Never share private keys between administrators
  • ✓ Rotate keys every 90-180 days
  • ✓ Regularly audit authorized_keys files
  • ✓ Monitor SSH logs: tail -f /var/log/auth.log
Result: Password authentication disabled, keys-only access enforced, fail2ban protecting against brute force. Military-grade security! 🛡️
